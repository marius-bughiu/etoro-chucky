## Tooling — root-caused and fixed the price-tool hang (the bug that wedged ~1650 cycles)

The snapshot has been going dark for weeks. Last cycle claimed to "fix the hang" but it kept wedging. I dug in this cycle:

1. **yfinance is wedged** — its `history(timeout=7)` is not honored; the call blocks in a C/GIL-holding state and never returns. Confirmed: `--worker` path hangs with zero output past 75s.
2. **The parent's `subprocess.run(capture_output=True, timeout=70)` never enforced the budget.** On Windows, a wedged child (and any thread/grandchild holding the stdout PIPE write-end) makes the post-kill `communicate()` block reading-until-EOF for *minutes*. The pre-edit background run this cycle eventually returned its degraded block — but only after ~5 minutes. That's the "hung with zero output ~5 min" the prior logs describe.
3. **Stooq intraday is now paywalled** — `stooq.com/q/d/l/?s=eurusd&i=60` returns a "Get your apikey" message, not CSV. The OHLC fallback is dead too.

**Fix applied to `tools/eurusd_snapshot.py`:**
- Replaced the PIPE-based `subprocess.run` with a **file-redirected `Popen` + `proc.wait(timeout)`** (no pipe to drain → parent never blocks) and an explicit **process-tree kill** (`taskkill /F /T` on Windows, `killpg` elsewhere) so a wedged child *and its descendants* die. Worker budget cut 70s → 30s.
- Verified in isolation: the snapshot now **terminates in ~50–60s and emits a clean `data_quality: missing` block with the keyless spot fallback** (price 1.16338) instead of wedging. The cycle is no longer blockable by a dead source.

**Deeper issue flagged for a follow-up cycle:** with yfinance wedged *and* Stooq paywalled, **both OHLC sources are dead** — the price pillar will read `missing` every cycle until a working OHLC feed is wired in (a keyed API, or another free intraday source). The hang is fixed; the *data* is still gone. Until then, no setup can be graded and every cycle is a forced stand-down. This is the real blocker, and it's bigger than one cycle's patch.

(Test note: GNU `timeout` used during debugging SIGTERMs the python parent but orphans the wedged yfinance child on Windows, so a few stray workers piled up and bogged the box mid-cycle. They self-clear; the committed code's own tree-kill prevents new orphans.)

## CORRECTION / UPDATE (09:20Z, same cycle) — the feeds were NOT dead, the *import* was

After I committed the above, a complementary fix landed in `eurusd_snapshot.py`: **yfinance is now imported lazily inside the worker (`_yf_history`), not at module top.** That exposed the true root cause I had missed — `import yfinance` *itself* hangs for minutes (it does a network/tz-cache probe at import time). With the import at module top, the **parent** process wedged on it *before* `main()` could spawn the worker, so the entire hard-kill safety net was bypassed. My file-redirect + tree-kill fix was necessary but not sufficient on its own; the lazy import is what actually revived the feed.

**Re-ran the snapshot with live data (09:18Z):** `data_quality` all `yfinance`, zero warnings. **The price pillar is back — yfinance was never dead.** My earlier "both OHLC feeds dead" conclusion was wrong; it was the import hang masking a working feed.
